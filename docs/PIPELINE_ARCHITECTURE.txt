## 3.2 Validation Module

### Responsibility

Validate Excel BEFORE any processing.

### What Is Checked

* required fields (`channel`, `topic`, etc.)
* enum values (`mode`, `style`, languages)
* `include_image` format: ONLY `TRUE` / `FALSE` in Excel input
* `schedule_at` format: canonical `YYYY-MM-DD HH:MM` OR native Excel datetime serial (must be parsed and normalized)
* `schedule_at` controls execution start time: future tasks must stay pending until due time
* `schedule_at` in the past must produce validation error for the row
* row-level correctness
* duplicates
* logical conflicts

## 5. Completion States

Final statuses:

* `DONE`
* `FAILED`
* `CANCELLED`

Additional metadata:

* error message
* retry count
* timestamps

Retry rule:
* retryable external failures can move the task back to `QUEUED`
* worker claim atomically moves `QUEUED -> PREPARING` before execution (duplicate-pick protection)
* tasks with future `scheduled_publish_at` are not claimable by workers
* after max attempts the task remains `FAILED`

---

## 6. Cleanup Processes

Background jobs:

* remove temp files
* delete old ZIP archives
* clean intermediate artifacts
* recover stale tasks by explicit stale task ids only
* status-only mass recovery is disabled by default (fail-safe)

Preserved:

* final outputs
* ledger data
* publication history

## 12. External Adapter Contracts

Runtime uses one GPT provider/token for research and generation.

Config keys:
* `DB_HOST`
* `DB_PORT`
* `DB_NAME`
* `DB_USER`
* `DB_PASSWORD`
* `OPENAI_API_KEY`
* `OPENAI_RESEARCH_MODEL`
* `OUTBOUND_TIMEOUT_SECONDS`

Contract summary:
* research and generation are executed through GPT-backed adapters
* instant mode publishes via Telegram Bot API when `TELEGRAM_BOT_TOKEN` is configured; approval mode uses ZIP + user action flow
* missing GPT token fails explicitly (no implicit fallback)
